从零开始Demo一个.Net Core WebAPI

Posted on 2021-05-01

随着.Net Core平台越来越成熟，特别是.Net 5开始要大一统，传统的.Net Framework渐渐要退出舞台，那么今天闲来无事，跟各位分享一下，从项目的构建开始写一个.Net Core的WebAPI Demo，尽量使用DDD思想和Restful。

第一部分，构建项目

在此之前，要安装最新版VS2019(16.9)，并安装.Net Core SDK 3.1或者.Net 5都可以。

  • 新建一个项目，选择模板为Asp.NetCore.WebAPI，去掉HTTS，启用OpenAPI(默认会加上swagger相关支持)。起名为Sam.NetCore.Demo.Hosting。

  • 干掉Hosting自带的Controller下的类和模型类

  • 新建项目.Net Core类库，起名Sam.NetCore.Demo.Domain，用于存放Entity实体和Repository抽象

  • 新建项目.Net Core类库，起名Sam.NetCore.Demo.DTO,用于存放DTO相关类
  • 新建项目.Net Core类库，起名Sam.NetCore.Demo.Application,用于Service相关
  • 新建项目.Net Core类库，起名Sam.NetCore.Demo.EntityFrameworkCore,用于EFCore的处理和Repository实现
  • 新建项目.Net Core类库，起名Sam.NetCore.Demo.EntityFrameworkCore.DBMigration,用于迁移数据库，当然也可以不建此层，迁移也在EntityFrameworkCore中操作。
  • 新建项目.Net Core类库，起名Sam.NetCore.Demo.Constant,用于存放常量和枚举
  • 新建项目.Net Core类库，起名Sam.NetCore.Demo.Utilities，用于存放公共方法和扩展

项目依赖关系如下

  • Hosting层依赖Application，Constant，DTO，EntityFrameworkCore
  • Application层依赖Contanst，Domain，DTO,EntityFrameworkCore,Utilities
  • Constant层先不添加项目依赖
  • Domain层先不添加项目依赖（后面有需要可以依赖Contanst）
  • DTO层先不添加项目依赖
  • EntityFrameworkCore层依赖Constant，Domain,Utilities
  • EntityFrameworkCore.DBMigration依赖EntityFrameworkCore
  • Utilities层先不添加依赖
    打开Hosting层的properties的launchSetting.json，修改一下，默认以控制台方式启动

至此，基本项目结构就构建完成了

主要是像IEmunable类型的返回类型，如果返回{value:[],links:[]}这种格式就有点违反restful返回资源的初衷，所以增加特定媒体类型来响应返回，例如application/vnd.mycompany.hateoas+json，如果请求这种格式就返回带hateoas的，否则就返回纯资源，以此区分。

缓存

客户端/浏览器缓存

应该使用session，cookie或者localstorage之类的东西

过期模型和验证模型

过期模型就是设置一个过期时间，只要没过期，直接从缓存返回。

验证模型就是Header添加一个例如ETag或者LastModifiedTime之类的属性，服务器验证这个属性，来决定是从缓存返回还是重新返回数据。

服务器缓存（过期模型）

最简单的缓存自带实现方法:

缓存标志：在ressponse header中设置，手动添加Cache-Contorl或者使用特性ResponseCache(Attribute)，只是设置Response Header，并没有实现缓存存储。

缓存存储：使用Response cache中间件

使用ETag（支持过期模型和验证模型）

简单的说，就是更新资源的时候ETag会改变，所以更新资源后再请求就会从服务器重新返回数据，否则就返回cache中的数据。

使用第三方库Marvin.Cache.Headers

请求时Header中添加If-None-Match，写入ETag值，服务器会判断

200代表重新返回数据，304代表没有修改，也就是从缓存取数据。

并发

举例

客户1get一个API，客户2get一个API，然后客户2 put了，然后客户1再put，这是客户1就会覆盖客户2的操作。

悲观和乐观并发控制
悲观并发控制

客户1请求时就锁定，锁定期间别人无法修改。 Restful API无法实现

乐观并发控制

客户1会得到一个token，只有token有效，就证明别人没更新过，那么客户1就可以更新，利用ETag
